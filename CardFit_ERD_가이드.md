# CardFit ERD 가이드

> 이 문서는 CardFit 프로젝트의 데이터베이스 구조(ERD)를 **처음 보는 사람도 이해할 수 있도록** 설명한 가이드입니다.

---

## 0. 먼저 알아둘 용어

| 용어                      | 쉬운 설명                                                  |
| ----------------------- | ------------------------------------------------------ |
| **DB (Database)**       | 데이터를 저장해두는 창고                                          |
| **테이블 (Table)**         | 창고 안의 서랍장 하나. 같은 종류의 데이터를 모아둠 (예: 회원 서랍, 카드 서랍)        |
| **행 (Row / Record)**    | 서랍 속 데이터 한 줄 (예: 회원 1명, 카드 1개)                         |
| **컬럼 (Column / Field)** | 각 데이터가 가진 항목 (예: 이메일, 이름, 가격)                          |
| **ERD**                 | Entity Relationship Diagram. 테이블들이 서로 어떻게 연결되는지 그린 설계도 |
| **PK (Primary Key)**    | 각 행을 고유하게 구별하는 번호표. 절대 중복되지 않음 (예: 회원 id)              |
| **FK (Foreign Key)**    | 다른 테이블의 PK를 가리키는 연결고리 (예: "이 게시글은 3번 회원이 썼다"의 3번)      |
| **1:N 관계**              | 하나가 여러 개를 가질 수 있는 관계 (예: 회원 1명 → 게시글 N개)               |

---

## 1. CardFit은 어떤 서비스인가요?

소비 내역을 분석해서 **나에게 가장 혜택이 큰 삼성카드를 추천**해주는 서비스입니다.
사용자는 소비 데이터(CSV)나 설문을 입력하고, AI가 그 데이터를 분석해 맞춤 카드를 추천합니다.
또한 카드 관련 정보를 나누는 커뮤니티 기능도 있습니다.

---

## 2. 테이블 한눈에 보기 (총 9개)

### 회원 / 소비 영역

| 테이블                      | 역할                                        |
| ------------------------ | ----------------------------------------- |
| **USER**                 | 회원 정보 (이메일, 비밀번호, 닉네임)                    |
| **USER_SURVEY**          | 회원의 소비 + 선호 설문 데이터 (식비, 교통비 등 카테고리별 월 지출) |
| **SPENDING_TRANSACTION** | 설문에 딸린 개별 거래 내역 (가맹점, 금액, 날짜)             |

### 카드 영역

| 테이블               | 역할                           |
| ----------------- | ---------------------------- |
| **CARD**          | 삼성카드 상품 정보 (카드사, 카드명, 연회비 등) |
| **CARD_BENEFIT**  | 각 카드가 제공하는 혜택 (식비 10% 할인 등)  |
| **CARD_WISHLIST** | 회원이 찜한 카드 목록                 |

### 커뮤니티 영역

| 테이블           | 역할      |
| ------------- | ------- |
| **POST**      | 게시글     |
| **COMMENT**   | 댓글      |
| **POST_LIKE** | 게시글 좋아요 |

---

## 3. 테이블 상세 설명

### USER (회원)

모든 데이터의 출발점입니다. 한 명의 회원이 설문도 쓰고, 카드도 찜하고, 게시글도 씁니다.

- `id` (PK): 회원 고유 번호
- `email`, `password_hash`, `username`: 로그인 정보

### USER_SURVEY (소비 설문)

회원이 입력한 소비 패턴입니다. CSV 업로드 또는 직접 입력으로 생성됩니다.

- `user_id` (FK → USER): 누구의 설문인지
- `food_monthly`, `transport_monthly` 등: 카테고리별 월 지출액
- `age_group`, `income_level`, `max_annual_fee`: 추천에 쓰이는 추가 정보

### SPENDING_TRANSACTION (거래 내역)

설문이 CSV로 만들어진 경우, 그 안의 개별 결제 건들이 여기 저장됩니다.

- `survey_id` (FK → USER_SURVEY): 어느 설문에 속한 거래인지
- `merchant`, `amount`, `category`, `transaction_date`: 가맹점, 금액, 분류, 날짜

### CARD (카드 상품)

추천 대상이 되는 카드들의 기본 정보입니다.

- `id` (PK): 카드 고유 번호
- `card_company`, `card_name`, `annual_fee`: 카드사, 이름, 연회비

### CARD_BENEFIT (카드 혜택)

한 카드는 여러 혜택을 가질 수 있어 별도 테이블로 분리했습니다.

- `card_id` (FK → CARD): 어느 카드의 혜택인지
- `benefit_category`, `discount_rate`, `monthly_limit`: 혜택 분류, 할인율, 월 한도

### CARD_WISHLIST (찜)

회원이 마음에 든 카드를 저장합니다.

- `user_id` (FK → USER): 누가 찜했는지
- `card_id` (FK → CARD): 어떤 카드를 찜했는지

### POST (게시글)

커뮤니티 게시글입니다.

- `user_id` (FK → USER): 작성자
- `title`, `content`, `category`, `views_count`: 제목, 내용, 분류, 조회수

### COMMENT (댓글)

게시글에 달리는 댓글입니다.

- `post_id` (FK → POST): 어느 게시글의 댓글인지
- `user_id` (FK → USER): 댓글 작성자

### POST_LIKE (좋아요)

게시글 좋아요 기록입니다.

- `post_id` (FK → POST): 어느 게시글에 대한 좋아요인지
- `user_id` (FK → USER): 누가 눌렀는지
- **제약조건**: 한 사람이 같은 글에 좋아요를 두 번 누를 수 없음 (post_id + user_id 조합은 유일해야 함)

## 4. 테이블 간 관계 (연결선 읽는 법)

ERD에서 선이 의미하는 것:

- **실선** = 필수 관계 (반드시 연결되어야 함)
- **점선** = 선택적 관계
- **1:N** = 왼쪽 하나가 오른쪽 여러 개를 가짐

| 관계                                   | 의미                 |
| ------------------------------------ | ------------------ |
| USER 1:N USER_SURVEY                 | 회원 1명이 여러 번 설문 가능  |
| USER_SURVEY 1:N SPENDING_TRANSACTION | 설문 1개에 거래 내역 여러 건  |
| USER 1:N CARD_WISHLIST               | 회원 1명이 카드 여러 개 찜   |
| CARD 1:N CARD_WISHLIST               | 카드 1개를 여러 회원이 찜    |
| CARD 1:N CARD_BENEFIT                | 카드 1개에 혜택 여러 개     |
| USER 1:N POST                        | 회원 1명이 게시글 여러 개 작성 |
| USER 1:N COMMENT                     | 회원 1명이 댓글 여러 개 작성  |
| USER 1:N POST_LIKE                   | 회원 1명이 좋아요 여러 개 누름 |
| POST 1:N COMMENT                     | 게시글 1개에 댓글 여러 개    |
| POST 1:N POST_LIKE                   | 게시글 1개에 좋아요 여러 개   |

> **왜 USER가 POST, COMMENT, POST_LIKE 3곳에 다 연결되나요?**
> 한 명의 회원이 게시글도 쓰고, 댓글도 달고, 좋아요도 누르기 때문입니다.
> 각 활동의 "누가 했는지"를 기록해야 하므로 세 테이블 모두 USER를 가리킵니다.

---

## 5. 실제 데이터 흐름 예시

처음 보는 사람이 가장 빨리 이해하는 방법은 "데이터가 어떻게 흐르는지" 따라가 보는 것입니다.

```
1. 회원가입        → USER 에 행 1개 생성
2. CSV 업로드      → USER_SURVEY 1개 + SPENDING_TRANSACTION 여러 개 생성
3. AI 카드 추천    → CARD + CARD_BENEFIT 데이터를 분석해 결과 반환
4. 마음에 든 카드  → CARD_WISHLIST 에 저장
5. 커뮤니티 글 작성 → POST 생성 → 다른 회원이 COMMENT, POST_LIKE 추가
```

---

## 6. 핵심 정리

- 모든 활동의 중심은 **USER**입니다. 거의 모든 테이블이 USER와 연결됩니다.
- **소비 데이터**(USER_SURVEY → SPENDING_TRANSACTION)와 **카드 데이터**(CARD → CARD_BENEFIT)가 AI 추천의 두 축입니다.
- **커뮤니티**(POST → COMMENT / POST_LIKE)는 나중에 추가된 기능으로, 회원 간 정보 공유를 담당합니다.
- **PK**는 각 행의 번호표, **FK**는 테이블을 잇는 연결고리라는 점만 기억하면 ERD 전체가 읽힙니다.
